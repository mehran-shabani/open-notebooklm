# دستورالعمل های امنیتی

این سند به تشریح شیوه های امنیتی برای توسعه نوت بوک باز می پردازد. از آسیب‌پذیری‌های واقعی کشف‌شده از طریق افشای هماهنگ با [CERT-EU] (https://cert.europa.eu) اطلاع‌رسانی می‌شود و باید به عنوان خواندن اجباری برای همه مشارکت‌کنندگان در نظر گرفته شود.

## گزارش آسیب پذیری ها

اگر آسیب‌پذیری امنیتی پیدا کردید، **مشکل عمومی GitHub را باز نکنید**. در عوض:

1. از [GitHub Security Advisories] (https://github.com/lfnovo/open-notebook/security/advisories/new) برای گزارش خصوصی استفاده کنید
2. یا مستقیماً به نگهبانان ایمیل بزنید

ما افشای آسیب‌پذیری هماهنگ را دنبال می‌کنیم و قبل از هر گونه اعلام عمومی برای رفع مشکل با شما کار خواهیم کرد.

## پرس و جوهای پایگاه داده (تزریق SurrealQL)

**قانون: هرگز ورودی کاربر را از طریق f-strings در جستارهای SurrealQL درون یابی نکنید.**

تزریق SurrealQL معادل تزریق SQL است. مقادیر کنترل‌شده توسط کاربر باید به‌عنوان متغیرهای پیوند پارامتری شده با استفاده از نحو «$variable» ارسال شوند.

### پرس و جوهای پارامتری (ایمن)

``پایتون
# خوب: پرس و جو پارامتری شده
نتیجه = منتظر repo_query(
    "انتخاب * از منبع WHERE id = $id"،
    {"id": sure_record_id(source_id)}
)
```

### درون یابی رشته F (آسیب پذیر)

``پایتون
# بد: ورودی کاربر در f-string
result = await repo_query (f"SELECT * FROM source WHERE id = {source_id}")
```

### ORDER BY و سایر بندهایی که نمی توان آنها را پارامتر کرد

«ORDER BY»، «LIMIT» و عبارت‌های مشابه معمولاً نمی‌توانند متغیرهای bind را در SurrealDB بپذیرند. به جای آن از ** اعتبار لیست مجاز ** استفاده کنید:

``پایتون
# خوب: در برابر لیست مجاز اعتبارسنجی کنید، سپس درون یابی کنید
allow_fields = {"name"، "created"، "updated"}
allow_directions = {"صعود"، "نزولی"}

Parts = order_by.strip().lower().split()
اگر قطعات[0] در allow_fields نباشد:
    raise HTTPException(status_code=400, detail="فیلد مرتب سازی نامعتبر")
اگر len(parts) > 1 and part[1] در allow_directions نباشد:
    raise HTTPException(status_code=400, detail="جهت مرتب سازی نامعتبر")

query = f"SELECT * FROM notebook ORDER BY {validated_order_by}"
```

برای اجرای مرجع اعتبارسنجی پارامتر مرتب سازی، به «api/routers/sources.py» مراجعه کنید.

### چک لیست

- [ ] همه مقادیر ارائه شده توسط کاربر از اتصال "$variable" استفاده می کنند
- [ ] هر رشته f در یک پرس و جو فقط حاوی مقادیر معتبر/هاردکد شده است
- [ ] «سفارش بر اساس»، «LIMIT»، و غیره از اعتبارسنجی لیست مجاز استفاده کنید
- [ ] مقادیر پایگاه داده مورد استفاده در جستارهای بعدی نیز پارامتری هستند (از تزریق مرتبه دوم جلوگیری می کند)

## ارائه الگو (تزریق قالب سمت سرور)**قانون: همیشه هنگام رندر قالب‌های Jinja2 که حاوی محتوای ارائه‌شده توسط کاربر هستند، از «SandboxedEnvironment» استفاده کنید.**

کتابخانه [ai-prompter](https://github.com/lfnovo/ai-prompter) (>= 0.4.0) به طور پیش‌فرض از «SandboxedEnvironment» استفاده می‌کند که دسترسی به ویژگی‌های خطرناک پایتون مانند «__globals__»، «__subclasses__» و «__init» را مسدود می‌کند.

### آنچه SandboxedEnvironment از آن جلوگیری می کند

```jinja2
{# اینها مسدود شده‌اند و خطای امنیتی را افزایش می‌دهند #}
{{ cycler.__init__.__globals__.os.popen('id').read() }}
{{ ''.__class__.__mro__[1].__subclasses__() }}
```

### دستورالعمل

- هرگز ai-prompter را زیر 0.4.0 تنزل ندهید
- در صورت استفاده مستقیم از Jinja2 (خارج از Ai-prompter)، همیشه از `jinja2.sandbox.SandboxedEnvironment` استفاده کنید
- هرگز رشته های ارائه شده توسط کاربر را مستقیماً به 'jinja2.Environment' یا 'jinja2.Template' منتقل نکنید

## مدیریت فایل (پیمایش مسیر و گنجاندن فایل محلی)

### آپلود فایل

**قانون: همیشه نام فایل ها را پاکسازی کنید و مسیرهای حل شده را تأیید کنید.**

``پایتون
واردات سیستم عامل
از مسیر واردات pathlib

# 1. نوار اجزای دایرکتوری
safe_filename = os.path.basename (نام_فایل اصلی)

# 2. اعتبار سنجی مسیر حل شده در فهرست هدف باقی می ماند
حل شد = (Path(upload_folder) / safe_filename).resolve()
اگر نه str(resolved).startswith(str(Path(upload_folder).resolve()) + os.sep):
    raise ValueError ("پیمایش مسیر شناسایی شد")
```

نکات کلیدی:

- از «os.path.basename()» برای حذف اجزای دایرکتوری از نام فایل های ارائه شده توسط کاربر استفاده کنید.
- از «Path.resolve()» برای حل و فصل symlinks و اجزای «..» استفاده کنید
- برای جلوگیری از دور زدن دایرکتوری خواهر و برادر (به عنوان مثال، «/uploads_evil/» مطابق با «/uploads») از «startswith()» استفاده کنید.

### ورودی های مسیر فایل

**قانون: تأیید کنید که هر مسیر فایل ارائه شده توسط کاربر در دایرکتوری مورد انتظار باشد.**

``پایتون
uploads_resolved = مسیر(UPLOADS_FOLDER).resolve()
file_resolved = مسیر(user_provided_path).resolve()
اگر نه str(file_resolved).startswith(str(uploads_resolved) + os.sep):
    raise HTTPException(status_code=400, detail="مسیر فایل نامعتبر")
```

هرگز مسیرهای فایل ارائه شده توسط کاربر را مستقیماً به عملکردهای خواندن فایل یا استخراج محتوا بدون اعتبار سنجی منتقل نکنید.

### چک لیست

- [ ] نام فایل های آپلود شده با «os.path.basename()» پاکسازی می شوند
- [ ] مسیرهای حل شده با «startswith(directory + os.sep)» اعتبارسنجی می شوند
- [ ] مقادیر «file_path» ارائه شده توسط کاربر قبل از استفاده تأیید می شوند
- [ ] بدون ایجاد دایرکتوری از ورودی کاربر ('mkdir' با مسیرهای پیمایش)

## احراز هویت و CORS

### احراز هویتOpen Notebook در حال حاضر از میان افزار ساده مبتنی بر رمز عبور («PasswordAuthMiddleware») استفاده می کند. این برای استقرارهای خود میزبانی تک کاربره مناسب است اما برای تولید باید سخت شود:

- رمز عبور پیش فرض را تغییر دهید ('OPEN_NOTEBOOK_PASSWORD')
- کلید رمزگذاری پیش فرض را تغییر دهید (`OPEN_NOTEBOOK_ENCRYPTION_KEY`)
- استقرار در پشت یک پروکسی معکوس با احراز هویت مناسب (OAuth، OIDC) را در نظر بگیرید.

### CORS

پیکربندی پیش‌فرض CORS به همه مبداها اجازه می‌دهد (`allow_origins=["*"]`). این برای بهبود در [#730] (https://github.com/lfnovo/open-notebook/issues/730) پیگیری می‌شود. برای استقرار تولید، مبدا را فقط به URL frontend محدود کنید.

## مدیریت اسرار

### کلید رمزگذاری

«OPEN_NOTEBOOK_ENCRYPTION_KEY» برای رمزگذاری کلیدهای API ذخیره شده در SurrealDB استفاده می‌شود. در حال تولید:

- تنظیم یک کلید قوی و منحصر به فرد (از پیش فرض استفاده نکنید)
- در صورت امکان از اسرار Docker از طریق «OPEN_NOTEBOOK_ENCRYPTION_KEY_FILE» استفاده کنید
- هرگز این مقدار را وارد یا افشا نکنید

### متغیرهای محیطی

- مقادیر حساس (کلیدهای API، گذرواژه‌ها، کلیدهای رمزگذاری) هرگز نباید در گزارش‌ها ظاهر شوند.
- از «loguru» با احتیاط استفاده کنید - از ثبت درخواست‌های کامل یا تخلیه‌های محیطی خودداری کنید
- کانتینر Docker به طور پیش فرض به عنوان root اجرا می شود. اجرا را به عنوان یک کاربر غیر ریشه در نظر بگیرید

## فهرست چک امنیتی مرور کد

هنگام بررسی روابط عمومی، موارد زیر را بررسی کنید:

1. **تزریق پرس و جو**: هر رشته f حاوی ورودی کاربر در کوئری SurrealQL
2. **تزریق قالب**: رشته های ارائه شده توسط کاربر بدون سندباکس به Jinja2 ارسال می شود
3. **پیمایش مسیر**: نام فایل ها یا مسیرهای ارائه شده توسط کاربر که بدون پاکسازی استفاده می شوند
4. ** افشای اطلاعات **: پیام های خطایی که مسیرهای داخلی، ردیابی پشته یا پیکربندی را نشان می دهد.
5. **SSRF**: URL های ارائه شده توسط کاربر بدون اعتبارسنجی به درخواست های HTTP سمت سرور ارسال می شود
6. **Secrets in log**: مقادیر حساس ثبت شده در هر سطح
