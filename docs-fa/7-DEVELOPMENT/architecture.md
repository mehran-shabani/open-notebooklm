# معماری نوت بوک را باز کنید

## بررسی اجمالی سطح بالا

دفترچه یادداشت باز از یک معماری سه لایه پیروی می کند که نگرانی ها را از هم جدا می کند:

```
┌────────────────── ──────────────────
│ مرورگر شما │
│ دسترسی: http://your-server-ip:8502 │
└────────────────┬ ───────────────────
                 │
                 ▼
         ┌───────────────┐
         │ پورت 8502 │ ← Next.js Frontend (آنچه می بینید)
         │ Frontend │ همچنین درخواست های API را به صورت داخلی پراکسی می کند!
         └───────┬───────┘
                 │ پروکسی /api/* درخواست ↓
                 ▼
         ┌───────────────┐
         │ پورت 5055 │ ← FastAPI Backend (به درخواست ها رسیدگی می کند)
         │ API │
         └───────┬───────┘
                 │
                 ▼
         ┌───────────────┐
         │ SurrealDB │ ← پایگاه داده (داخلی، پیکربندی خودکار)
         │ (پورت 8000) │
         └───────────────┘
```

**نکات کلیدی:**
- **v1.1+**: Next.js به طور خودکار درخواست های «/api/*» را به باطن پراکسی می کند و راه اندازی معکوس پروکسی را ساده می کند
- مرورگر شما فرانت اند را از پورت 8502 بارگیری می کند
- فرانت اند باید بداند که API را کجا پیدا کند - هنگام دسترسی از راه دور، تنظیم کنید: `API_URL=http://your-server-ip:5055`
- **پشت پراکسی معکوس؟** اکنون فقط باید به پورت 8502 پروکسی کنید! [Reverse Proxy Configuration] (../5-CONFIGURATION/reverse-proxy.md) را ببینید

## معماری تفصیلی

Open Notebook بر اساس معماری **سه لایه و اولین ناهمگام** ساخته شده است که برای مقیاس پذیری، مدولار بودن و انعطاف پذیری هوش مصنوعی چند ارائه دهنده طراحی شده است. این سیستم نگرانی‌ها را در لایه‌های frontend، API و پایگاه داده از هم جدا می‌کند، با LangGraph که جریان‌های کاری هوشمند را تقویت می‌کند و اسپرانتو یکپارچه‌سازی یکپارچه با ارائه‌دهندگان 8+ AI را ممکن می‌سازد.

**فلسفه اصلی**:
- اول حریم خصوصی: کاربران داده ها و انتخاب ارائه دهنده هوش مصنوعی خود را کنترل می کنند
- Async/wait در سراسر: عملیات غیر مسدود برای UX پاسخگو
- طراحی دامنه محور: تفکیک واضح بین مدل های دامنه، مخازن، و ارکستراتورها
- انعطاف پذیری چند ارائه دهنده: ارائه دهندگان هوش مصنوعی را بدون تغییر کد برنامه تعویض کنید
- قابلیت میزبانی خودکار: همه اجزای قابل استقرار در محیط های ایزوله

## معماری سه لایه

### لایه 1: Frontend (React/Next.js @ port 3000)

**هدف**: رابط کاربری پاسخگو و تعاملی برای تحقیق، یادداشت ها، چت و مدیریت پادکست.

**پشته فناوری**:
- **Framework**: Next.js 15 با React 19- **زبان**: TypeScript با بررسی دقیق نوع
- ** مدیریت ایالت**: Zustand (فروشگاه سبک وزن) + TanStack Query (وضعیت سرور)
- **استایل**: Tailwind CSS + Shadcn/ui کتابخانه مؤلفه
- **ابزار ساخت**: بسته وب (همراه با Next.js)

**مسئولیت های کلیدی**:
- نوت بوک ها، منابع، یادداشت ها، جلسات چت و پادکست ها را رندر کنید
- مدیریت تعاملات کاربر (عملیات ایجاد، خواندن، به روز رسانی، حذف)
- مدیریت وضعیت UI پیچیده (مدال ها، آپلود فایل ها، جستجوی بلادرنگ)
- پاسخ های جریانی از API (چت، تولید پادکست)
- نمایش جاسازی ها، نتایج جستجوی برداری، و بینش

**الگوی ارتباط**:
- همه داده‌ها از طریق REST API واکشی شده‌اند (درخواست‌های همگام‌سازی با پورت 5055)
- URL پایه پیکربندی شده: «http://localhost:5055» (dev) یا محیط خاص (prod)
- TanStack Query ذخیره، واکشی مجدد و همگام سازی داده ها را کنترل می کند
- Zustand وضعیت جهانی را ذخیره می کند (کاربر، نوت بوک، زمینه انتخاب شده)
- CORS در سمت API برای درخواست های متقاطع فعال است

**معماری کامپوننت**:
- `/src/app/`: روتر برنامه Next.js (صفحات، طرح‌بندی‌ها)
- `/src/components/`: اجزای React قابل استفاده مجدد (دکمه ها، فرم ها، کارت ها)
- `/src/hooks/`: قلاب های سفارشی (useNotebook، useChat، useSearch)
- `/src/lib/`: توابع ابزار، کلاینت های API، اعتباردهنده ها
- `/src/styles/`: CSS جهانی، پیکربندی Tailwind

### لایه 2: API (FastAPI @ پورت 5055)

**هدف**: عملیات‌های باطنی با آرامش در نوت‌بوک‌ها، منابع، یادداشت‌ها، جلسات چت و مدل‌های هوش مصنوعی.

**پشته فناوری**:
- **فریم ورک**: FastAPI 0.104+ (چارچوب وب پایتون ناهمگام)
- **زبان**: Python 3.11+
- ** اعتبار سنجی **: Pydantic v2 (طرحواره های درخواست/پاسخ)
- **Logging**: Loguru (ورود به سیستم JSON ساختاریافته)
- **تست **: Pytest (تست های واحد و ادغام)

**معماری**:
```
برنامه FastAPI (main.py)
  ├── روترها (نقاط پایانی HTTP)
  │ ├── routers/notebooks.py (عملیات CRUD)
  │ ├── routers/sources.py (حذف محتوا، آپلود)
  │ ├── routers/notes.py (مدیریت یادداشت)
  │ ├── routers/chat.py (جلسات مکالمه)
  │ ├── روتر/search.py (متن کامل + جستجوی برداری)
  │ ├── routers/transformations.py (تبدیل های سفارشی)
  │ ├── routers/models.py (پیکربندی مدل AI)
  │ └── روتر/*.py (11 روتر اضافی)
  │
  ├── خدمات (منطق تجاری)
  │ ├── *_service.py (ارکستراسیون، فراخوانی گراف)
  │ ├── command_service.py (ارسال کار ناهمگام)
  │ └── میان افزار (Auth، Logging)
  │
  ├── مدل ها (طرحواره های پیدانتیک)
  │ └── models.py (اعتبارسنجی، سریال سازی)
  │
  └── طول عمر (راه اندازی/خاموش)└── AsyncMigrationManager (مهاجرت های طرحواره پایگاه داده)
```

**مسئولیت های کلیدی**:
1. **رابط HTTP**: درخواست های REST را بپذیرید، اعتبارسنجی کنید، پاسخ های JSON را برگردانید
2. **منطق تجاری**: مدل های دامنه، عملیات مخزن و گردش کار را هماهنگ کنید
3. **صف کار غیرهمگام**: کارهای طولانی مدت (تولید پادکست، پردازش منبع) را ارسال کنید.
4. ** مهاجرت های پایگاه داده **: به روز رسانی طرحواره را در هنگام راه اندازی اجرا کنید
5. **Error Handling**: استثناها را بگیرید، کدهای وضعیت HTTP مناسب را برگردانید
6. ** ثبت نام **: عملیات برای اشکال زدایی و نظارت را دنبال کنید

**جریان راه اندازی **:
1. متغیرهای محیطی «.env» را بارگیری کنید
2. برنامه FastAPI را با میان افزار CORS + auth راه اندازی کنید
3. AsyncMigrationManager را اجرا کنید (شما پایگاه داده را ایجاد/به روز می کند)
4. ثبت همه روترها (بیش از 20 نقطه پایانی)
5. سرور آماده روی پورت 5055

**چرخه درخواست-پاسخ**:
```
درخواست HTTP → روتر → سرویس → دامنه/مخزن → SurrealDB
                                       ↓
                                  LangGraph (اختیاری)
                                       ↓
پاسخ ← سریال سازی پیدانتیک ← سرویس ← نتیجه
```

### لایه 3: پایگاه داده (SurrealDB @ port 8000)

**هدف**: پایگاه داده نمودار با جاسازی های برداری داخلی، جستجوی معنایی و مدیریت روابط.

**پشته فناوری**:
- **پایگاه داده**: SurrealDB (تراکنش های چند مدلی، ACID)
- **زبان پرس و جو**: SurrealQL (سینتکس SQL مانند با عملیات گراف)
- **Async Driver**: مشتری Async Rust برای پایتون
- **Migrations**: فایل های دستی «.surql» در «/migrations/» (اجرای خودکار در راه اندازی API)

**جدول اصلی**:

| جدول | هدف | فیلدهای کلیدی |
|-------|---------|-----------|
| "نوت بوک" | ظرف پروژه تحقیقاتی | شناسه، نام، توضیحات، بایگانی شده، ایجاد شده، به روز شده |
| "منبع" | مورد محتوا (PDF، URL، متن) | شناسه، عنوان، متن کامل، موضوعات، دارایی، ایجاد شده، به روز شده |
| "منبع_جاسازی" | جاسازی های برداری برای جستجوی معنایی | شناسه، منبع، جاسازی، chunk_text، chunk_index |
| "یادداشت" | یادداشت های تحقیقاتی ایجاد شده توسط کاربر | شناسه، عنوان، محتوا، note_type (human/ai)، ایجاد شده، به روز شده |
| "جلسه_چت" | جلسه گفتگو | شناسه، notebook_id، عنوان، پیام ها (JSON)، ایجاد شده، به روز شده |
| "تحول" | قوانین تبدیل سفارشی | شناسه، نام، توضیحات، درخواست، ایجاد، به روز رسانی |
| "منبع_بینش" | خروجی تبدیل | id، source_id، insight_type، محتوا، ایجاد شده، به روز شده |
| "مرجع" | رابطه: منبع → دفترچه یادداشت | خارج (منبع)، در (یادداشت) |
| "مصنوع" | رابطه: یادداشت → دفترچه یادداشت | خارج (یادداشت)، در (یادداشت) |**نمودار رابطه**:
```
دفترچه یادداشت
  ↓ (مرجع_توسط)
منبع
  ├→ SourceEmbedding (1: بسیاری برای متن تکه تکه شده)
  ├→ SourceInsight (1: بسیاری برای خروجی های تبدیل)
  └← توجه (از طریق رابطه مصنوع)
    ├← جاسازی (جستجوی معنایی)
    └→ موضوعات (برچسب ها)

جلسه گفتگو
  ├← دفترچه یادداشت
  └← پیام ها (ذخیره شده به عنوان آرایه JSON)
```

**قابلیت جستجوی برداری**:
- جاسازی ها به صورت بومی در SurrealDB ذخیره می شوند
- جستجوی متن کامل در «source.full_text» و «note.content».
- جستجوی شباهت کسینوس در بردارهای تعبیه شده
- جستجوی معنایی با نقطه پایانی جستجو ادغام می شود

**مدیریت اتصال**:
- ترکیب اتصال ناهمگام (اندازه قابل تنظیم)
- پشتیبانی از تراکنش برای عملیات چند رکوردی
- اعتبار سنجی خودکار طرحواره از طریق مهاجرت
- حفاظت از زمان پرس و جو (جلوگیری از پرس و جوهای بی نهایت)

## منطق پشته فناوری

### چرا Python + FastAPI؟

**پایتون**:
- اکوسیستم غنی AI/ML (LangChain، LangGraph، ترانسفورماتورها، scikit-learn)
- نمونه سازی و استقرار سریع
- پشتیبانی گسترده async (asyncio، async/wait)
- نکات نوع قوی (Pydantic، mypy)

**FastAPI**:
- فریمورک مدرن، اول ناهمگام
- اسناد OpenAPI خودکار (Swagger UI @ /docs)
- اعتبار سنجی درخواست داخلی (Pydantic)
- عملکرد عالی (معیاری نزدیک به سرعت C/Rust)
- تزریق میان افزار/وابستگی آسان

### چرا Next.js + React + TypeScript؟

**Next.js**:
- چارچوب React تمام پشته با SSR/SSG
- مسیریابی مبتنی بر فایل (ساختار پروژه بصری)
- مسیرهای API داخلی (هم‌مکانی اختیاری)
- تقسیم تصویر/کد بهینه شده
- استقرار آسان (Vercel، Docker، خود میزبانی)

**React 19**:
- رابط کاربری مبتنی بر مؤلفه (قابل استفاده مجدد، قابل آزمایش)
- ابزار و جامعه عالی
- مدیریت دولتی سمت مشتری (Zustand)
- همگام سازی حالت سمت سرور (TanStack Query)

**TypeScript**:
- تایپ خطاهای امنیتی در زمان کامپایل
- تکمیل خودکار IDE و refactoring بهتر
- مستندسازی از طریق انواع (کد خود مستندسازی)
- ورود آسان تر برای مشارکت کنندگان جدید

### چرا SurrealDB؟

**SurrealDB**:
- پایگاه داده گراف بومی (روابط درجه یک هستند)
- تعبیه‌های برداری داخلی (بدون DB برداری جداگانه)
- تراکنش های ACID (ثبات داده ها)
- چند مدل (رابطه ای + سند + نمودار)
- جستجوی متن کامل + جستجوی معنایی در یک پرس و جو
- میزبانی خود (برخلاف Pinecone/Weaviate مدیریت شده)
- SurrealQL انعطاف پذیر (SQL-Syntax)

**جایگزین در نظر گرفته شده**: PostgreSQL + pgvector (افزونه های بالغ تر اما مجزا)

### چرا اسپرانتو برای ارائه دهندگان هوش مصنوعی؟

**کتابخانه اسپرانتو**:- رابط یکپارچه برای بیش از 8 ارائه دهنده LLM (OpenAI، Anthropic، Google، Groq، Ollama، Mistral، DeepSeek، xAI)
- تعبیه‌های چند ارائه‌دهنده (OpenAI، Google، Ollama، Mistral، Voyage)
- ادغام TTS/STT (OpenAI، Groq، ElevenLabs، Google)
- انتخاب هوشمند ارائه دهنده (منطق بازگشتی، بهینه سازی هزینه)
- پشتیبانی از نادیده گرفتن مدل هر درخواست
- پشتیبانی محلی Ollama (گزینه کاملاً خود میزبان)

**جایگزین در نظر گرفته شده**: انتزاع ارائه دهنده LangChain (پرمخاطب تر، کمتر انعطاف پذیر)

## گردش کار LangGraph

LangGraph یک کتابخانه ماشین دولتی است که جریان های کاری هوش مصنوعی چند مرحله ای را هماهنگ می کند. Open Notebook از پنج گردش کار اصلی استفاده می کند:

### 1. **روند کار پردازش منبع** ('open_notebook/graphs/source.py')

**هدف**: دریافت محتوا (PDF، URL، متن) و آماده شدن برای جستجو/بینش.

**جریان**:
```
ورودی (فایل/URL/متن)
  ↓
استخراج محتوا (کتابخانه هسته محتوا)
  ↓
متن را پاک و نشانه گذاری کنید
  ↓
ایجاد جاسازی (اسپرانتو)
  ↓
سوابق SourceEmbedding را ایجاد کنید (قطع شده + نمایه شده)
  ↓
استخراج موضوعات (خلاصه LLM)
  ↓
در SurrealDB ذخیره کنید
  ↓
خروجی (سوابق منبع با جاسازی ها)
```

**دستورالعمل ایالتی**:
``پایتون
{
  "content_state": {"file_path" | "url" | "content": str},
  "source_id": خیابان،
  "full_text": خیابان،
  "Embeddings": فهرست[Dict]،
  "موضوعات": فهرست[خ]،
  "notebook_ids": List[str],
}
```

** فراخوان شده توسط**: Sources API ("POST /sources")

---

### 2. **جریان کاری چت** ('open_notebook/graphs/chat.py')

**هدف**: مکالمات چند نوبتی با مدل هوش مصنوعی، ارجاع به زمینه نوت بوک انجام دهید.

**جریان**:
```
پیام کاربر
  ↓
زمینه ساخت (منابع / یادداشت های انتخاب شده)
  ↓
افزودن پیام به جلسه
  ↓
ایجاد اعلان چت (سیستم + تاریخچه + زمینه)
  ↓
تماس با LLM (از طریق اسپرانتو)
  ↓
پاسخ جریان
  ↓
پیام هوش مصنوعی را در ChatSession ذخیره کنید
  ↓
خروجی (پیام کامل)
```

**دستورالعمل ایالتی**:
``پایتون
{
  "session_id": str,
  "پیام ها": فهرست [BaseMessage]،
  "context": Dict[str، Any]، # منبع، یادداشت‌ها، قطعه‌ها
  "پاسخ": خیابان،
  "model_override": اختیاری[str]،
}
```

**ویژگی های کلیدی**:
- سابقه پیام در SurrealDB (ایست بازرسی SqliteSaver) ادامه داشت
- ساخت زمینه از طریق ابزار «build_context_for_chat()».
- شمارش رمز برای جلوگیری از سرریز
- پشتیبانی از نادیده گرفتن مدل در هر پیام

** فراخوان شده توسط **: Chat API (`POST /chat/execute`)

---

### 3. **Ask Workflow** ('open_notebook/graphs/ask.py')

**هدف**: با جستجو در منابع و ترکیب پاسخ ها به سؤالات کاربر پاسخ دهید.

**جریان**:
```
سوال کاربر
  ↓
استراتژی جستجوی طرح (LLM جستجوها را ایجاد می کند)
  ↓اجرای جستجوها (بردار + جستجوی متن)
  ↓
نتایج امتیاز و رتبه
  ↓
ارائه پاسخ (LLM از نتایج ترکیب می شود)
  ↓
پاسخ های جریانی
  ↓
خروجی (پاسخ نهایی)
```

**دستورالعمل ایالتی**:
``پایتون
{
  "سوال": خیابان،
  "استراتژی": استراتژی جستجو،
  "answers": List[str],
  "final_answer": str,
  "sources_used": فهرست[منبع]،
}
```

## الگوهای طراحی

### 1. **طراحی دامنه محور (DDD)**

**اشیاء دامنه** (`Open_notebook/domain/`):
- "یادداشت": ظرف تحقیق با روابط با منابع/یادداشت ها
- «منبع»: مورد محتوا (PDF، URL، متن) با جاسازی‌ها
- "توجه": یادداشت تحقیقاتی ایجاد شده توسط کاربر یا هوش مصنوعی
- 'ChatSession': تاریخچه مکالمه برای یک نوت بوک
- "تغییر": قانون سفارشی برای استخراج بینش

**الگوی مخزن**:
- لایه دسترسی به پایگاه داده (`open_notebook/database/repository.py`)
- `repo_query()`: کوئری های SurrealQL را اجرا کنید
- `repo_create()`: درج رکوردها
- `repo_upsert()`: ادغام رکوردها
- `repo_delete()`: حذف رکوردها

**روشهای موجودیت**:
``پایتون
```

### 2. **Async-First Architecture**

**همه I/O ناهمگام است**:
- درخواست های پایگاه داده: `wait repo_query(...)`
- LLM فراخوانی می کند: «await model.ainvoke(...)».
- ورودی/خروجی فایل: «wait upload_file.read()».
- فراخوان های نمودار: `wait graph.ainvoke(...)`

**مزایا**:
- رسیدگی به درخواست بدون مسدود کردن (FastAPI چندین درخواست همزمان را ارائه می دهد)
- استفاده بهتر از منابع (انتظار I/O CPU را مسدود نمی کند)
- تناسب طبیعی برای syntax async/wait Python

**مثال**:
``پایتون
@router.post("/sources")

```
خدمات، اشیاء دامنه، مخازن، و گردش کار را هماهنگ می کند:

``پایتون
```

### اضافه کردن یک مدل داده جدید

1. مدل را در «open_notebook/domain/model_name.py» ایجاد کنید
2. ارث بردن از BaseModel (ابژه دامنه)
3. متدهای «save()»، «get()»، «delete()» (CRUD) را پیاده سازی کنید.
4. در صورت نیاز به پرس و جوهای پیچیده، توابع مخزن را اضافه کنید
5. مهاجرت پایگاه داده را در `migrations/` ایجاد کنید
6. مسیرها و مدل های API را در `api/` اضافه کنید

### اضافه کردن یک ارائه دهنده جدید هوش مصنوعی

1. اسپرانتو را برای ارائه دهنده جدید پیکربندی کنید (به .env.example مراجعه کنید)
2. ModelManager به طور خودکار از طریق متغیرهای محیطی شناسایی می کند
3. لغو از طریق پیکربندی هر درخواست (بدون نیاز به تغییر کد)
4. منطق برگشتی را در صورت در دسترس نبودن ارائه دهنده آزمایش کنید

## ملاحظات استقرار

### توسعه

- تمام خدمات در لوکال هاست (3000، 5055، 8000)
- بارگذاری مجدد خودکار در تغییرات فایل (Next.js، FastAPI)
- انتقال پایگاه داده با بارگذاری مجدد داغ
- اسناد API را در http://localhost:5055/docs باز کنید

### تولید

- **Frontend**: در Vercel، Netlify یا Docker مستقر شوید
- **API**: ظرف داکر (به Dockerfile مراجعه کنید)
- **پایگاه داده**: ظرف SurrealDB یا سرویس مدیریت شده
- **محیط**: فایل .env را با کلیدهای API ایمن کنید
- **SSL/TLS**: پروکسی معکوس (Nginx، CloudFlare)
- **محدودیت نرخ**: در لایه پروکسی اضافه کنید
- **Auth**: جایگزینی PasswordAuthMiddleware با OAuth/JWT
- **مانیتورینگ**: تجمیع گزارش (CloudWatch، DataDog و غیره)

## خلاصهمعماری Open Notebook یک پایه محکم برای تحقیقات مبتنی بر حریم خصوصی و مبتنی بر هوش مصنوعی فراهم می کند. جداسازی نگرانی ها (frontend/API/پایگاه داده)، طراحی async-first، و انعطاف پذیری چند ارائه دهنده امکان توسعه سریع و استقرار آسان را فراهم می کند. گردش کار LangGraph وظایف پیچیده هوش مصنوعی را هماهنگ می کند، در حالی که اسپرانتو جزئیات ارائه دهنده را خلاصه می کند. نتیجه یک سیستم مقیاس پذیر و قابل نگهداری است که کاربران را در کنترل داده ها و انتخاب ارائه دهنده هوش مصنوعی خود قرار می دهد.
