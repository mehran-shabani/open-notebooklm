# اصول طراحی و چشم انداز پروژه

این سند به تشریح اصول اصلی، چشم انداز و فلسفه طراحی می پردازد که توسعه Notebook را هدایت می کند. همه مشارکت‌کنندگان باید قبل از پیشنهاد تغییرات یا ویژگی‌های جدید، این اصول را بخوانند و درک کنند.

## 🎯 پروژه چشم انداز

Open Notebook می‌خواهد جایگزینی برای Notebook LM Google ** متمرکز بر حریم خصوصی و میزبانی خود باشد که به کاربران این امکان را می‌دهد:

1. ** داده های تحقیقاتی خود را داشته باشید ** - کنترل کامل بر محل زندگی داده ها و افرادی که می توانند به آنها دسترسی داشته باشند
2. **ارائه دهندگان هوش مصنوعی خود را انتخاب کنید** - آزادی استفاده از هر ارائه دهنده هوش مصنوعی یا اجرای مدل های محلی
3. **جریان کاری خود را سفارشی کنید** - انعطاف پذیری برای انطباق ابزار با نیازهای مختلف تحقیقاتی
4. **دسترسی به کار آنها در هر کجا ** - از طریق وب UI، API، یا ادغام

### نوت بوک باز چیست

- یک **دستیار پژوهش** برای مدیریت و درک محتوا
- یک **پلتفرم** که ارائه دهندگان مختلف هوش مصنوعی را به هم متصل می کند
- یک ابزار **اول برای حفظ حریم خصوصی** که داده های شما را تحت کنترل شما نگه می دارد
- یک **سیستم توسعه پذیر** با API و گزینه های سفارشی سازی

### آنچه که نوت بوک باز نیست

- یک ویرایشگر سند (از Google Docs، Notion و غیره برای آن استفاده کنید)
- یک سیستم ذخیره سازی فایل (از Dropbox، S3 و غیره برای آن استفاده کنید)
- یک ربات چت همه منظوره (از ChatGPT، Claude و غیره برای آن استفاده کنید)
- جایگزینی برای کل گردش کار شما (این یک ابزار در جعبه ابزار شما است)

## 🏗️ اصول طراحی اصلی

### 1. ابتدا حریم خصوصی

**اصل **: داده ها و تحقیقات کاربر باید به طور پیش فرض تحت کنترل کاربر باقی بماند.

**در عمل**:
- استقرار خود میزبانی مورد استفاده اولیه است
- بدون تله متری یا تجزیه و تحلیل بدون انتخاب صریح
- عدم وابستگی سخت به خدمات ابری خاص
- پاک کردن اسناد و مدارک در مورد اینکه چه داده‌هایی به کجا می‌روند

**تصمیمات نمونه**:
- ✅ پشتیبانی از مدل های محلی Ollama
- ✅ انتخاب ارائه دهنده هوش مصنوعی قابل تنظیم
- ❌ ادغام خدمات ابری با کد سخت
- ❌ وابستگی های خدمات خارجی مورد نیاز

### 2. سادگی بیش از ویژگی ها

**اصل **: ابزار باید به راحتی قابل درک و استفاده باشد، حتی اگر به معنای ویژگی های کمتر باشد.

**در عمل**:
- رابط کاربری واضح و متمرکز با بخش های کاملاً مشخص
- پیش فرض های معقولی که برای اکثر کاربران کار می کند
- ویژگی های پیشرفته پنهان در پشت پیکربندی اختیاری
- اسناد نوشته شده برای کاربران غیر فنی

**تصمیمات نمونه**:
- طرح بندی سه ستونی (منابع، یادداشت ها، چت)
- ✅ مدل‌های پیش‌فرض که خارج از جعبه کار می‌کنند
- ❌ کاربران غافلگیر کننده با گزینه های بیش از حد از قبل
- ❌ گردش کار چند مرحله ای پیچیده برای کارهای اساسی

### 3. API-First Architecture**اصل **: همه عملکردها باید از طریق API قابل دسترسی باشند، نه فقط UI.

**در عمل**:
- UI همان API را فراخوانی می کند که مشتریان خارجی از آن استفاده می کنند
- جامع REST API با اسناد OpenAPI
- هیچ ویژگی "فقط UI" که نمی تواند خودکار باشد
- جدایی واضح بین جلو و باطن

**تصمیمات نمونه**:
- ✅ باطن FastAPI با اسناد کامل API
- ✅ الگوهای API ثابت در تمام نقاط پایانی
- ❌ منطق تجاری در اجزای UI
- ❌ ویژگی هایی که نیاز به دسترسی مستقیم به پایگاه داده دارند

### 4. انعطاف پذیری چند ارائه دهنده

**اصل **: کاربران هرگز نباید در یک ارائه دهنده هوش مصنوعی قفل شوند.

**در عمل**:
- پشتیبانی از چندین ارائه دهنده هوش مصنوعی از طریق کتابخانه اسپرانتو
- تعویض آسان بین ارائه دهندگان و مدل ها
- پاک کردن اسناد در مورد محدودیت های ارائه دهنده
- تنزل برازنده زمانی که ارائه دهندگان در دسترس نیستند

**تصمیمات نمونه**:
- ✅ پشتیبانی از 18+ ارائه دهندگان هوش مصنوعی
- ✅ انتخاب مدل برای هر ویژگی (چت، جاسازی، TTS)
- ❌ ویژگی هایی که فقط با OpenAI کار می کنند
- ❌ نقاط پایانی API با کد سخت برای ارائه دهندگان خاص

### 5. توسعه پذیری از طریق استانداردها

**اصل **: سیستم باید از طریق رابط های کاملاً تعریف شده قابل گسترش باشد، نه از طریق فورکینگ.

**در عمل**:
- سیستم های پلاگین برای تبدیل ها و دستورات
- فرمت های استاندارد داده (JSON، Markdown)
- پاک کردن نقاط پسوند در معماری
- اسناد برای سناریوهای سفارشی سازی رایج

**تصمیمات نمونه**:
- ✅ قالب های تبدیل سفارشی
- ✅ سیستم فرمان پس زمینه
- ✅ قالب های سریع Jinja2
- ❌ منطق کسب و کار سخت کد شده بدون امتیاز پسوند

### 6. Async-First for Performance

**اصل **: عملیات طولانی مدت نباید رابط کاربری یا API را مسدود کند.

**در عمل**:
- الگوهای ناهمگام/انتظار در سراسر باطن
- پردازش کار پس زمینه برای کارهای سنگین
- به روز رسانی وضعیت و ردیابی پیشرفت
- مدیریت برازنده پاسخ های کند ارائه دهنده هوش مصنوعی

**تصمیمات نمونه**:
- ✅ AsyncIO برای عملیات پایگاه داده
- ✅ دستورات پس زمینه برای تولید پادکست
- ✅ پاسخ های جریانی برای چت
- ❌ عملیات مسدودسازی همزمان در نقاط پایانی API

## 🎨 اصول UI/UX

### روی محتوا تمرکز کنید نه کروم

- به هم ریختگی و حواس پرتی UI را به حداقل برسانید
- محتوا باید بیشتر فضای صفحه را اشغال کند
- کنترل ها در صورت نیاز ظاهر می شوند، همیشه قابل مشاهده نیستند
- طرح‌بندی ثابت در نماهای مختلف

### افشای پیشرونده

- ابتدا گزینه های ساده، گزینه های پیشرفته در صورت تقاضا را نشان دهید
- کاربران جدید را با هر تنظیمات ممکن غرق نکنید- پیش فرض های معقولی را ارائه دهید که برای 80٪ موارد استفاده کار می کند
- ویژگی های قدرت را قابل کشف اما نه مزاحم کنید

### پاسخگو و سریع

- UI باید برای عملیات معمول احساس فوری داشته باشد
- نمایش حالت های بارگیری برای عملیات هایی که زمان می برد
- کش و بهینه سازی در صورت امکان
- در اتصالات آهسته به زیبایی تنزل دهید

## 🔧 اصول فنی

### جداسازی خالص نگرانی ها

**لایه ها نباید نشت کنند**:
- فرانت اند نباید از ساختار پایگاه داده اطلاع داشته باشد
- API نباید حاوی منطق تجاری باشد (تخصیص به لایه دامنه)
- مدل های دامنه نباید از درخواست های HTTP اطلاع داشته باشند
- لایه پایگاه داده نباید در مورد ارائه دهندگان هوش مصنوعی بداند

### نوع Safety and Validation

**اشتباهات را زود تشخیص دهید**:
- از مدل های Pydantic برای تمام مرزهای API استفاده کنید
- نکات را در سرتاسر پایگاه کد پایتون تایپ کنید
- TypeScript برای کدهای ظاهری
- اعتبارسنجی داده ها در مرزهای سیستم

### آنچه مهم است را آزمایش کنید

**تمرکز روی تست های ارزشمند**:
- منطق کسب و کار و مدل های دامنه را تست کنید
- تست قراردادهای API و مدیریت خطا
- کد چارچوب (FastAPI، React و غیره) را آزمایش نکنید.
- تست های یکپارچه سازی برای گردش های کاری حیاتی

### پایگاه داده به عنوان منبع حقیقت

** SurrealDB تنها منبع حقیقت ماست**:
- همه حالت در پایگاه داده باقی ماند
- بدون منطق تجاری در لایه پایگاه داده
- از ویژگی های SurrealDB (پیوندها، پرس و جوها) به درستی استفاده کنید
- مهاجرت های طرحواره برای همه تغییرات طرحواره

- آیا استفاده و درک آن ساده است؟

## 🔄 تکامل اصول

این اصول در سنگ بنا نیست. همانطور که پروژه رشد می کند و ما از کاربران یاد می گیریم، برخی از اصول ممکن است تکامل یابند. با این حال، تغییرات در اصول اصلی باید به شرح زیر باشد:

1. **توجیه شده** - دلیل روشنی برای اینکه چرا نیاز به تغییر است
2. **بحث آزادانه** - ورودی جامعه در مورد تغییرات عمده
3. **مستند ** - به روز شده در این سند با توضیحات
4. **تدریج ** - به عنوان تغییرات شکستن در صورت امکان اجرا نمی شود
