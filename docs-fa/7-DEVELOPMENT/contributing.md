# مشارکت در Open Notebook

از علاقه شما به مشارکت در Open Notebook سپاسگزاریم! ما از مشارکت توسعه دهندگان در تمام سطوح مهارت استقبال می کنیم. این راهنما به شما کمک می کند تا گردش کار مشارکت ما را درک کنید و اینکه چه چیزی باعث مشارکت خوب می شود.

## 🚨 گردش کار شماره اول

** برای حفظ انسجام پروژه و جلوگیری از هدر رفتن تلاش، لطفاً این روند را دنبال کنید:**

1. **ابتدا مشکلی ایجاد کنید** - قبل از نوشتن هر کدی، مشکلی را ایجاد کنید که باگ یا ویژگی را توصیف کند.
2. **راه حل خود را پیشنهاد دهید** - نحوه برنامه ریزی برای پیاده سازی اصلاح یا ویژگی را توضیح دهید
3. **منتظر تعیین تکلیف باشید** - یک نگهدارنده در صورت تایید موضوع را بررسی و به شما اختصاص خواهد داد.
4. **فقط پس از آن شروع به کدنویسی کنید** - این اطمینان حاصل می کند که کار شما با چشم انداز و معماری پروژه هماهنگ است.

**چرا این روند؟**
- از انجام کارهای تکراری جلوگیری می کند
- اطمینان حاصل می کند که راه حل ها با اصول معماری و طراحی ما مطابقت دارند
- با دریافت بازخورد قبل از کدنویسی در وقت شما صرفه جویی می کند
- به نگهبانان کمک می کند تا جهت پروژه را مدیریت کنند

> ⚠️ ** درخواست های کششی بدون مشکل تعیین شده ممکن است بسته شود **، حتی اگر کد خوب باشد. ما می‌خواهیم با اطمینان از همسو بودن کار قبل از شروع، به وقت شما احترام بگذاریم.

## کد رفتار

با شرکت در این پروژه، از شما انتظار می رود که از آیین نامه رفتار ما حمایت کنید. محترمانه، سازنده و همکار باشید.

## چگونه می توانم کمک کنم؟

### گزارش اشکالات

1. **جستجوی مسائل موجود ** - بررسی کنید که آیا اشکال قبلا گزارش شده است
2. **یک گزارش اشکال ایجاد کنید** - از [الگوی گزارش اشکال] استفاده کنید (https://github.com/lfnovo/open-notebook/issues/new?template=bug_report.yml)
3. ** جزئیات را ارائه دهید ** - شامل:
   - مراحل تکثیر
   - رفتار مورد انتظار در مقابل رفتار واقعی
   - گزارش ها، اسکرین شات ها یا پیام های خطا
   - محیط شما (سیستم عامل، نسخه داکر، نسخه نوت بوک باز)
4. **مشخص کنید که آیا می‌خواهید آن را برطرف کنید** - اگر علاقه دارید کادر "من می‌خواهم روی این کار کنم" را علامت بزنید.

### ویژگی های پیشنهادی

1. **جستجوی مسائل موجود ** - بررسی کنید که آیا این ویژگی قبلاً پیشنهاد شده است یا خیر
2. **یک درخواست ویژگی ایجاد کنید** - از [قالب درخواست ویژگی] استفاده کنید (https://github.com/lfnovo/open-notebook/issues/new?template=feature_request.yml)
3. **مقدار را توضیح دهید** - توضیح دهید که چرا این ویژگی مفید است
4. **پیشنهاد اجرای ** - اگر ایده هایی در مورد نحوه اجرای آن دارید، آنها را به اشتراک بگذارید
5. **مشخص کنید که آیا می خواهید آن را بسازید** - اگر علاقه دارید کادر "من می خواهم روی این کار کنم" را علامت بزنید

### کد مشارکت (درخواست‌های کششی)

**مهم: قبل از شروع هر گونه روابط عمومی، گردش کار شماره اول بالا را دنبال کنید**پس از تعیین مشکل شما:

1. ** مخزن را فورک کنید** و شعبه خود را از «اصلی» ایجاد کنید
2. **دیدگاه و اصول ما را درک کنید** - [design-principles.md](design-principles.md) را بخوانید تا بفهمید چه چیزی تصمیمات ما را هدایت می کند.
3. **معماری ما را دنبال کنید** - برای درک ساختار پروژه به مستندات معماری مراجعه کنید
4. **کد کیفیت بنویسید** - استانداردهای ذکر شده در [code-standards.md](code-standards.md) را دنبال کنید.
5. **تغییرات خود را آزمایش کنید** - برای دستورالعمل های تست به [testing.md](testing.md) مراجعه کنید
6. **به روز رسانی اسناد ** - اگر عملکرد را تغییر دادید، اسناد مربوطه را به روز کنید
7. ** روابط عمومی خود را ایجاد کنید **:
   - به شماره شماره ارجاع دهید (به عنوان مثال، "اصلاحات #123")
   - توضیح دهید که چه چیزی و چرا تغییر کرده است
   - شامل اسکرین شات برای تغییرات UI
   - روابط عمومی را متمرکز نگه دارید - یک موضوع در هر روابط عمومی

### چه چیزی باعث مشارکت خوب می شود؟

✅ **ما عاشق روابط عمومی هستیم که:**
- حل یک مشکل واقعی که در یک شماره توضیح داده شده است
- استانداردهای معماری و کدنویسی ما را دنبال کنید
- شامل تست ها و مستندات
- دارای محدوده مناسب (تمرکز بر یک چیز)
- پیام های تعهد واضح داشته باشید

❌ ** ممکن است روابط عمومی را ببندیم که:**
- مشکل تایید شده مرتبطی نداشته باشید
- تغییرات قطعی را بدون بحث و گفتگو معرفی کنید
- تعارض با دیدگاه معماری ما
- فقدان آزمایش یا مستندات
- سعی کنید چندین مشکل نامرتبط را حل کنید

## پیام های Git Commit

- از زمان حال استفاده کنید ("Add feature" نه "Added feature")
- از حالت امری استفاده کنید ("انتقال مکان نما به..." نه "انتقال مکان نما به...")
- خط اول را به 72 کاراکتر یا کمتر محدود کنید
- ارجاع به مسائل و کشیدن درخواست ها آزادانه پس از خط اول

## گردش کار توسعه

### استراتژی شعبه

ما از ** گردش کار شاخه ویژگی** استفاده می کنیم:

1. **شعبه اصلی**: `اصلی` - کد آماده تولید
2. ** شاخه های ویژگی **: `ویژگی/توضیح` - ویژگی های جدید
3. ** رفع اشکال **: `رفع/توضیح` - رفع اشکال
4. **مستندات**: "اسناد/توضیحات" - به روز رسانی اسناد

### ایجاد تغییرات

1. **یک شاخه ویژگی ایجاد کنید**:
``باش
git checkout -b feature/amazing-new-feature
```

2. **تغییرات خود را** طبق استانداردهای کدگذاری ما انجام دهید

3. **تغییرات خود را آزمایش کنید**:
``باش
# تست ها را اجرا کنید
uv run pytest

# لینتینگ را اجرا کنید
uv run ruff check .

# قالب بندی را اجرا کنید
فرمت uv run ruff .
```

4. **تغییرات خود را انجام دهید**:
``باش
git افزودن .
git commit -m "feat: افزودن ویژگی جدید شگفت انگیز"
```

5. ** فشار و ایجاد روابط عمومی **:
``باش
ویژگی git push origin/ویژگی شگفت انگیز-جدید
# سپس یک Pull Request در GitHub ایجاد کنید
```

### چنگال خود را به روز نگه دارید

``باش
# تغییرات بالادستی را واکشی کنید
git fetch upstream

# تغییر به اصلی و ادغامgit checkout main
git ادغام upstream/main

# به چنگال خود فشار دهید
git push source main
```

## فرآیند درخواست کشش

هنگامی که یک درخواست کشش ایجاد می کنید:

1. **مشکل خود را پیوند دهید** - به شماره شماره در توضیحات روابط عمومی مراجعه کنید
2. **تغییرات خود را شرح دهید** - توضیح دهید که چه چیزی تغییر کرده و چرا
3. **ارائه شواهد آزمون** - اسکرین شات، نتایج آزمایش، یا گزارش
4. **الگوی روابط عمومی را بررسی کنید** - مطمئن شوید که تمام بخش های مورد نیاز را تکمیل کرده اید
5. **منتظر بررسی باشید** - یک نگهدارنده روابط عمومی شما را ظرف یک هفته بررسی می کند

### انتظارات بررسی روابط عمومی

- بازخورد بررسی کد مربوط به کد است، نه شخص
- پذیرای پیشنهادات و رویکردهای جایگزین باشید
- آدرس نظرات بررسی با وضوح و احترام
- اگر بازخورد نامشخص است، سؤال بپرسید
